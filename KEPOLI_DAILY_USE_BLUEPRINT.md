# Kepoli — Daily-Use Completion Blueprint

> Market-benchmarked, agent-executable plan to finish the super-app (restaurant + delivery + courier) to genuine daily-use.
> 6 areas benchmarked - 58 candidate items - 43 planned. Built 2026-06-21.

## Definition of "daily-use done"

**Restaurant:** An owner can run a full Friday-night rush AND a returning diner can reorder in seconds, both without friction. OWNER bar (Toast/Square/Foodics parity for the daily loop): orders auto-accept with a default quote time (no manual-confirm tax) with manual-approve still opt-in; busy-mode/snooze with timed auto-resume so ordering is never silently left off; a real cash drawer (opening float + pay-in/out log + blind count → over/short) so the Z-report is a true end-of-day close; KDS shows an all-day aggregate prep count and scheduled orders surface as 'due soon'; a solo owner has a single next-action focus surface; stale-pending orders escalate via push even off-screen. WAITER bar (Toast Go/TouchBistro parity): seat-level ordering + split-the-bill BY SEAT (not just ÷N); a glanceable floor map; course assignment + hold/send-new at entry; quick-keys for top sellers; tip capture on settle; active push (buzz/badge) for waiter-calls and order-ready; share/print the bill. CUSTOMER bar (Uber Eats/Deliveroo parity): pre-order ETA on the menu header AND at checkout; availability-safe 1-tap reorder unified into one code path; edit a cart line in place; 'complete your order' upsell + favorites/popular rail; express checkout that remembers fulfillment/address/payment; server-side order history so reorder survives device switches. DONE = every item P0/P1 across these three customer-facing areas shipped, gate-green, and owner-validated on a real device.

**Delivery:** A driver does 20 drops one-handed on a propped, awake screen and never hunts for the next tap. DONE bar (Uber Driver/DoorDash Dasher parity): the active job is a sticky single-next-action hero with the advance button and the correct-leg Maps link pinned to the top, all settings/earnings/cash-out chrome demoted behind the fold; a new offer is a full-screen takeover with sound + haptic + a shrinking countdown ring so a time-boxed exclusive offer is unmissable; proof-of-delivery photo capture on the delivered step (wiring the already-built backend proof_photo_url); a driver-facing stats card (completion rate, rating, lifetime drops); distance-to-pickup on delivery offers (parity with ride offers); a GPS heartbeat so a stationary 'online' driver stays rankable plus a stale-GPS warning; screen wake-lock during an active job. DONE = all P0/P1 shipped, the dashboard fits one working screen, gate-green, and the glanceability/one-handed restructure validated on a real device (cannot be previewed here).

**Courier:** Send a same-city parcel and the recipient gets an Uber-Connect-grade experience, with repeat sends down to a couple of taps. DONE bar (Uber Connect/Glovo parity): saved Home/Work addresses (reusing the existing SavedAddress model) in BOTH pickup and drop-off pickers + recent recipients for 1-tap repeat sends; a sender-side pickup handover with a distinct 'collected' milestone (and correct package status vocabulary that separates 'courier arriving for pickup' from 'package collected / en route') instead of the courier self-advancing with no proof; a recipient tracking link + SMS so the person waiting sees the courier, the ETA, and reads the handover code off their OWN screen; package size/vehicle class (moto vs car) driving price + courier-fit + dispatch eligibility; split pickup vs drop-off instructions with leave-at-door/meet-at-curb intent. DONE = all P0/P1 shipped, gate-green, and the new recipient surface + handover behavior-change owner-validated on a real device.

## Agent-execution strategy

CONFLICT-SAFE AGENT-WAVE PROTOCOL. Run one wave at a time; within a parallelizable wave, spawn one agent PER FILE-GROUP so no two concurrent agents ever open the same file. Each agent owns its file-group end-to-end (backend model+migration+view, frontend page/component, tests) EXCEPT shared i18n files, which are never touched by feature agents (see i18n_bottleneck_handling).

PER-AGENT CONTRACT: (1) implement only your file-group's item(s); (2) add/extend unit tests for new backend endpoints/migrations and any non-trivial frontend logic; (3) return, as structured data, the list of NEW i18n keys with en/fr/ar values you require — do NOT edit messages-*.js yourself; (4) self-run the gates that don't need the aggregator.

PER-WAVE SEQUENCE (orchestrator-driven):
1. Run feature agents in parallel (disjoint files only). Each commits NOTHING yet — returns a diff + its i18n-key payload.
2. i18n CONSOLIDATION STEP: a single dedicated writer agent merges every agent's returned keys into messages-en.js / -fr.js / -ar.js and the messages.js aggregator in one pass, keeping the three locales key-aligned.
3. GATE VERIFICATION on the merged tree, in order, fail-closed:
   - Frontend (run from C:\Users\user\resto\frontend): `npm run verify:i18n` (catches any missing/mismatched locale key from step 2) then `npm run lint` (eslint --max-warnings=0) then `npm run build` then `npm run test` (vitest, target 142).
   - Backend (PowerShell tool, system interpreter): `$env:DJANGO_DEBUG='True'; C:\Python312\python.exe -m pytest` from C:\Users\user\resto\backend, target 4307 passed / 0 failed (the ~25 Postgres-needing errors are known-not-regressions; do not treat as failures). DJANGO_DEBUG=True is mandatory or SECRET_KEY hard-fails.
4. Only when ALL gates are green: `git checkout -b wave-<n>-<slug>` (repo is not the cwd default; branch off main, never commit to default), commit per file-group with co-author trailer, push.
5. TESTED-BUILD HANDOFF: produce a `npm run build` artifact + a backend migration-applied build and hand to the owner with a per-wave validation script (the concrete on-device taps to confirm). Block the next wave until the owner signs off — MANDATORY for any item risk=visual-restructure-needs-device-test or behavior-change, since those cannot be previewed here and only a real device reveals one-handed/thumb-reach/RTL regressions.

ORDERING RULE BAKED INTO WAVES: additive/logic wins (low risk, high daily-frequency) ship first; backend-migration items are grouped so migrations are linear and never concurrent on the same app (menu vs tenancy vs accounts migrations can run in parallel ACROSS apps but are serialized WITHIN an app); visual-restructures and behavior-changes ship last and always gated by an owner device-validation. If an agent's gate fails, the orchestrator reverts only that file-group's diff (others already merged stay), and re-dispatches a fix agent — the disjoint-file rule guarantees a clean partial revert.

ROLLBACK/RESUME: each wave is an isolated branch; a failed wave never blocks a shipped one. Migrations are additive-only (new nullable fields / new models) so a wave can be reverted without destructive down-migrations.

## i18n bottleneck handling

HARD RULE: feature agents NEVER edit messages-en.js, messages-fr.js, messages-ar.js, or the messages.js aggregator. These four files are a shared bottleneck — parallel edits guarantee merge collisions and locale drift.

MECHANISM (keys-as-data, single writer): every feature agent that needs copy returns its required keys as STRUCTURED DATA — a flat list of {keyPath, en, fr, ar} (e.g. {keyPath:'menu.etaReadyIn', en:'Ready in ~{min}–{max} min', fr:'Prêt dans ~{min}–{max} min', ar:'جاهز خلال ~{min}–{max} دقيقة'}). The agent references the key in its Vue/JS code via t('...') as if it exists, but does not create it.

CONSOLIDATION: after all feature agents in a wave return, ONE dedicated i18n-writer agent (the only agent that touches these files that wave) merges every returned key into all three locale files plus the aggregator in a single serialized pass, preserving key ordering/nesting and guaranteeing all three locales carry every new key (no orphan keys, no missing-locale). This runs as step 2 of every wave, BEFORE gate verification.

ENFORCEMENT: `npm run verify:i18n` (which runs verify-i18n.mjs + verify-i18n-usage.mjs) is the fail-closed gate — it catches (a) a key used in code but absent from a locale, (b) a key present in one locale but missing in another, (c) an unused key. The wave cannot pass gates if the consolidation missed or misaligned any key, so the single-writer step is mechanically verified, not trusted. RTL (Arabic) interpolation/ordering correctness for any new visual copy is additionally validated on-device in the wave's owner handoff.

WHY NOT PER-AGENT LOCKING: serializing i18n into one consolidation step is strictly simpler and collision-free versus file-locking, and it lets all feature agents run fully parallel since their only shared resource (copy) is deferred to a downstream single writer.

### Cross-cutting — super-app identity, onboarding, role-switching, notifications, real-time, wallet, daily-habit/retention
_vs Careem (super app — single identity, city-aware service grid, Careem Pay wallet as moat), Grab (GrabPay stored-value wallet as retention moat, one-tap home services), Gojek (multi-service home, GoPay, service tiles + 'orders in progress' rail), Uber / Uber Eats (real-time live tracking, iOS Live Activities / Dynamic Island, GPS-tied push, single-next-action driver app), Deliveroo / DoorDash (1-tap reorder, order-again rail, scheduled re-engagement push)_

**Verdict:** Kepoli already has the hard structural foundations that most super-app clones lack: ONE customer identity (single customer_id session) spanning food/shops/pharmacy/rides/courier/driver; a single closed-loop wallet with an auditable atomic ledger (top-up via Stripe + QR pay-code + voucher + P2P), a per-vertical activity/spend index, a robust auto-reconnecting WebSocket layer, and locale-aware web push with per-vertical + global opt-outs and a NotificationLog audit trail. That is roughly 70% of the way to the Careem/Grab bar. The headline gaps are all on the DAILY-HABIT surface, not the plumbing: (1) the super-app hub is a static launcher — it has the 'continue where you left off' DATA (count + last_activity per vertical) but never renders a resume card, greeting, or time-of-day/contextual ordering, so there is no reason to open it daily; (2) there is NO in-app notification center/inbox — notifications are fire-and-forget web push only, so a missed push is lost forever (Careem/Grab all keep a persistent inbox); (3) push permission is buried and never primed, so opt-in rates will be low and the whole real-time loop silently fails; (4) no global 'live order/ride' status bar that persists across vertical switches (the Uber Live-Activity equivalent) — the live banner only exists inside the per-tenant CustomerLayout, not the hub/marketplace; (5) wallet has no low-balance nudge or auto-reload, the single strongest retention mechanic Grab cites; (6) role-switching works but is a plain nav link, not a labelled identity switcher, and the hub doesn't reflect the user's driver/owner roles. Closing these is mostly additive frontend work on top of endpoints and data that already exist.
